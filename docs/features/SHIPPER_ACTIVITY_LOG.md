# Shipper Activity Log

## Overview

The Shipper Activity Log provides administrators with visibility into configuration change history and an audit trail for each log shipper. This feature helps track when changes were made, what actions occurred, and provides accountability for shipper management.

## Location

**Frontend**: `/frontend/src/views/Shippers.vue`
**Backend API**: `GET /api/shippers/:id/activity`

The Activity Log is displayed in the **View Shipper Dialog**, after the Volume Mounts section.

## Features

### Activity Types

The system tracks the following activity types:

| Activity Type | Description | Tag Color |
|--------------|-------------|-----------|
| `created` | Shipper was created | Success (green) |
| `config_updated` | Shipper configuration was updated | Primary (blue) |
| `source_added` | New log source was added | Success (green) |
| `source_updated` | Log source was modified | Warning (yellow) |
| `source_deleted` | Log source was removed | Danger (red) |
| `volume_added` | Volume mount was added | Success (green) |
| `volume_deleted` | Volume mount was removed | Danger (red) |
| `key_regenerated` | API key was regenerated | Warning (yellow) |

### Display Format

The activity log displays three columns:

1. **Timestamp**: When the activity occurred (formatted as "MMM dd, yyyy HH:mm")
2. **Action**: The activity type with color-coded tag
3. **Description**: Human-readable description of what happened

### Data Retention

- Activity log entries are stored in the `shipper_activity` database table
- The frontend displays the **50 most recent** activity records per shipper
- Activity records are retained indefinitely (no automatic cleanup)
- When a shipper is deleted, all associated activity records are cascade-deleted

## API Integration

### Endpoint

```
GET /api/shippers/:id/activity?limit=50
```

### Request Parameters

- `id` (path): Shipper ID (number)
- `limit` (query): Maximum number of records to return (default: 50)


## Code Implementation

### State Management

```typescript
const activityLog = ref<any[]>([]);
const activityLoading = ref(false);
```

### Fetch Function

```typescript
async function fetchShipperActivity(shipperId: number) {
  activityLoading.value = true;
  try {
    const response = await api.getShipperActivity(shipperId, 50);
    activityLog.value = response.data;
  } catch (error) {
    console.error('Failed to fetch activity log:', error);
    activityLog.value = [];
  } finally {
    activityLoading.value = false;
  }
}
```

### Helper Functions

#### Format Activity Type Display

```typescript
function formatActivityType(type: string): string {
  return type
    .split('_')
    .map(word => word.charAt(0).toUpperCase() + word.slice(1))
    .join(' ');
}

// Examples:
// 'created' → 'Created'
// 'config_updated' → 'Config Updated'
// 'key_regenerated' → 'Key Regenerated'
```

#### Get Activity Tag Color

```typescript
function getActivityType(type: string): string {
  const types: Record<string, string> = {
    created: 'success',
    config_updated: 'primary',
    source_added: 'success',
    source_updated: 'warning',
    source_deleted: 'danger',
    volume_added: 'success',
    volume_deleted: 'danger',
    key_regenerated: 'warning',
  };
  return types[type] || 'info';
}
```

## User Workflow

1. **Navigate to Shippers page** (`/shippers`)
2. **Click "View" button** on any shipper
3. **View Shipper Dialog opens** showing:
   - Shipper details
   - Log sources table
   - Volume mounts table
   - **Activity Log table** (automatically loaded)
4. **Review activity history**:
   - See when shipper was created
   - Track configuration changes
   - Monitor source additions/removals
   - Check API key regenerations
5. **Close dialog** when done

## Security Considerations

### Access Control

- Activity log is only accessible to authenticated users with shipper viewing permissions
- Backend enforces authorization on the `/api/shippers/:id/activity` endpoint
- No sensitive data (like full API keys) is stored in activity messages

### Audit Trail Benefits

1. **Accountability**: Track who made changes (when user metadata is included)
2. **Troubleshooting**: Correlate configuration changes with log ingestion issues
3. **Compliance**: Maintain records of system modifications
4. **Security Monitoring**: Detect unauthorized API key regenerations

#### Add CSV Export

```typescript
function exportActivityLog() {
  const csv = activityLog.value.map(row =>
    `"${formatDate(row.created_at)}","${row.activity_type}","${row.message}"`
  ).join('\n');

  const header = 'Timestamp,Activity Type,Description\n';
  const blob = new Blob([header + csv], { type: 'text/csv' });
  const url = window.URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `shipper-${currentShipper.value.id}-activity.csv`;
  a.click();
}
```

## Database Schema

### `shipper_activity` Table

```sql
CREATE TABLE shipper_activity (
  id SERIAL PRIMARY KEY,
  shipper_id INTEGER NOT NULL REFERENCES log_shippers(id) ON DELETE CASCADE,
  activity_type VARCHAR(50) NOT NULL,
  message TEXT NOT NULL,
  metadata JSONB,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_shipper_activity_shipper_id ON shipper_activity(shipper_id);
CREATE INDEX idx_shipper_activity_created_at ON shipper_activity(created_at DESC);
```

## Troubleshooting

### Activity Log Not Appearing

**Symptom**: Activity log section is empty even after shipper operations

**Causes**:
1. Backend not recording activity (check shipper controller code)
2. Database query failing (check backend logs)
3. Frontend API call failing (check browser console)

**Resolution**:
1. Verify backend endpoint: `curl http://localhost:5000/api/shippers/1/activity`
2. Check backend logs for errors
3. Verify database table exists and has records: `SELECT * FROM shipper_activity WHERE shipper_id = 1;`
4. Check browser console for JavaScript errors

### Loading State Stuck

**Symptom**: Activity log shows "Loading..." indefinitely

**Causes**:
1. API request hanging
2. Network timeout
3. Backend endpoint not responding

**Resolution**:
1. Check browser Network tab for failed requests
2. Verify backend is running: `docker ps`
3. Check backend logs: `docker logs siembox-backend`
4. Clear browser cache and reload

### Incorrect Tag Colors

**Symptom**: Activity types show wrong color tags

**Cause**: `getActivityType()` function missing mapping for activity type

**Resolution**:
1. Add missing activity type to `getActivityType()` mapping
2. Ensure backend uses consistent activity type strings
3. Update documentation with new activity types
